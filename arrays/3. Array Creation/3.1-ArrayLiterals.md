# Chapter 3 — Array Creation

## Phase 3.1 — Array Literal (`[]`)

---

# Goal

Understand **what the JavaScript engine actually does** when it encounters an array literal.

Example:

```javascript
const arr = [10, 20, 30];
```

This chapter is **not** about syntax.

It is about **how the engine manufactures an Array object** inside memory.

---

# Step 0 — Parsing Phase

Before execution begins, the parser reads:

```javascript
const arr = [10, 20, 30];
```

The parser immediately recognizes:

* `arr` → Variable declaration
* `[10, 20, 30]` → **Array Literal**

This is important.

The parser **does not** think:

```javascript
new Array(10,20,30)
```

Array literals and the `Array()` constructor are **different language constructs** even though both eventually produce Array objects.

---

# Step 1 — Creation Phase

Environment Record

```
arr
↓

<uninitialized>
```

Nothing has been created inside memory yet.

---

# Step 2 — Execution Phase Begins

Execution reaches

```javascript
[10,20,30]
```

The engine immediately allocates a brand-new **Array object**.

Conceptually:

```
Reference #A
↓

(Array Object)

↓

{

    length : 0

}
```

Notice:

The engine **does not** create

```
{}
```

and later convert it into an array.

It directly allocates an **Array object**.

---

# Important Observation

An Array object **always owns a `length` property**.

Therefore even an empty array starts life as

```
Reference #A
↓

{

    length : 0

}
```

The engine initializes this automatically.

You never wrote

```javascript
length : 0
```

The engine did.

---

# Step 3 — Reading Array Literal Elements

The engine now reads the literal **from left to right**.

Example:

```javascript
const arr = [10,20,30];
```

---

## Read First Element

Engine reads

```
10
```

Stores it at

```
Property "0"
```

Immediately afterwards

```
Reference #A
↓

{

    "0" : 10,

    length : 1

}
```

Notice something extremely important.

The engine **immediately updates the length**.

It does **not** wait until the entire array literal has finished.

---

## Read Second Element

Engine reads

```
20
```

Stores

```
Property "1"
```

Now

```
Reference #A
↓

{

    "0" : 10,

    "1" : 20,

    length : 2

}
```

---

## Read Third Element

Engine reads

```
30
```

Stores

```
Property "2"
```

Now

```
Reference #A
↓

{

    "0" : 10,

    "1" : 20,

    "2" : 30,

    length : 3

}
```

Execution of the array literal is complete.

Finally,

```
Reference #A
```

is returned and assigned to

```
arr
```

---

# How Does the Engine Decide the Length?

The engine follows one invariant:

> **`length` is always one greater than the highest numeric index.**

Examples:

```
Highest Index

0

↓

length = 1
```

```
Highest Index

2

↓

length = 3
```

```
Highest Index

10

↓

length = 11
```

The engine is **not counting elements**.

It simply tracks the highest numeric index.

---

# Example

```javascript
const arr = [10,20,30];

arr[5] = 100;
```

Conceptually

Before

```
Reference #A
↓

{

    "0" : 10,

    "1" : 20,

    "2" : 30,

    length : 3

}
```

After

```
Reference #A
↓

{

    "0" : 10,

    "1" : 20,

    "2" : 30,

    "5" : 100,

    length : 6

}
```

Notice:

The engine **does not create**

```
"3" : undefined
```

or

```
"4" : undefined
```

Those properties simply **do not exist**.

---

# Holes

Indexes 3 and 4 above are called **holes (empty slots)**.

They are **not** the same as

```javascript
undefined
```

Conceptually

```
{

    "0" : 10,

    "1" : 20,

    "2" : 30,

    "5" : 100,

    length : 6

}
```

Only property `"5"` exists.

Properties `"3"` and `"4"` were never created.

---

# Why This Matters

This single concept explains:

* `Array(5)`
* Sparse Arrays
* `slice()`
* Spread (`...`)
* `Array.from()`
* `push()`
* `pop()`
* `splice()`
* Iteration behavior
* `0 in arr`

Nearly every array method relies on understanding **holes vs actual properties**.

---

# Mental Model

Always visualize arrays as

```
Reference
↓

(Array Object)

↓

{

    "0" : value,

    "1" : value,

    ...

    length : N
}
```

Do **not** imagine arrays as magic contiguous memory.

Conceptually they are Array objects that own indexed properties and a special `length` property.

---

# Key Takeaways

✅ `[]` is Array Literal syntax.

✅ Parser recognizes it separately from `Array()`.

✅ Engine allocates an **Array object**, not an ordinary object.

✅ Every Array object starts with

```
length : 0
```

✅ Elements are inserted one by one from left to right.

✅ `length` is updated immediately after each insertion.

✅ `length = highest numeric index + 1`.

✅ Missing indexes are **holes**, not `undefined`.

✅ Assigning to a far index (e.g. `arr[5]`) creates only that property and updates `length`; intermediate indexes remain holes.
